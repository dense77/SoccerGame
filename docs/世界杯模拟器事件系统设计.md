# 世界杯模拟器事件系统设计

## 1. 文档目标

本文档定义首版事件系统的结构、触发机制、执行流程和测试基线，用于支撑比赛中的关键事件回合制体验。

首版事件系统只覆盖以下四类事件：

- 临时战术安排
- 特殊换人事件
- 球员需求事件
- 裁判争议事件

## 2. 设计目标

- 让比赛不只是静态数值结算
- 让玩家在关键节点做出有限但有影响的选择
- 让事件模板与具体执行解耦
- 让事件系统可以脱离 UI 独立运行
- 让事件效果能被比赛引擎解释并转换为修正值

## 3. 系统边界

事件系统负责：

- 根据比赛上下文筛选可触发事件
- 计算候选事件权重
- 生成事件文本和选项
- 解析选项效果
- 将效果写回比赛上下文

事件系统不负责：

- 直接计算最终赛果
- 直接更新积分榜
- 直接操作数据库连接
- 直接渲染 React 界面

## 4. 核心概念

### 4.1 Event Template

描述一个可复用的事件模板，包含：

- 事件类别
- 触发阶段
- 触发条件
- 文本模板
- 基础权重
- 重复规则
- 可选项列表
- 连锁事件信息

### 4.2 Event Option

描述玩家可做的一个选择，包含：

- 展示文案
- 效果集合
- 风险收益倾向
- 是否影响后续事件权重

### 4.3 Event Effect

描述某个选项对比赛上下文产生的变化。首版效果只允许修改“中间态”，不允许直接写死最终比分。

效果可作用于：

- 我方士气
- 对方士气
- 我方体能修正
- 对方体能修正
- 我方进攻修正
- 我方防守修正
- 阵型克制修正
- 战术相性修正
- 某类后续事件权重
- 某球员情绪或出场优先级

### 4.4 Match Context

事件系统读取和写回的运行态对象，至少包含：

- 当前阶段
- 当前比分
- 双方基础实力摘要
- 双方阵型与战术
- 双方士气与体能修正
- 已触发事件列表
- 玩家历史选择
- 球员状态摘要

## 5. 事件分类设计

### 5.1 临时战术安排

用途：

- 在落后、僵持或领先保守时给玩家战术微调权

典型事件：

- 压上争取进球
- 降低节奏稳住局面
- 集中边路冲击
- 加强中路保护

默认影响方向：

- 调整攻防修正
- 调整体能消耗
- 调整后续换人类事件权重

### 5.2 特殊换人事件

用途：

- 通过换人决策体现阵容深度和临场判断

典型事件：

- 核心球员体能下降
- 边锋状态火热值得加快换人
- 后腰背黄牌存在风险
- 前锋连续错失机会需要调整

默认影响方向：

- 替换特定球员
- 更新阵容修正
- 更新球员体能和情绪

### 5.3 球员需求事件

用途：

- 提供最小更衣室管理体验

典型事件：

- 核心球员要求增加球权
- 队长要求收缩防线
- 替补希望提前上场

默认影响方向：

- 调整球员情绪
- 调整某类战术修正
- 调整后续事件权重

### 5.4 裁判争议事件

用途：

- 提供高风险高波动的决策节点

典型事件：

- 对争议判罚激烈抗议
- 向第四官员施压
- 稳住情绪避免额外风险

默认影响方向：

- 调整士气
- 调整情绪稳定度
- 调整比赛风险修正

## 6. 事件模板结构

首版事件模板统一使用以下逻辑字段：

- `id`
- `code`
- `category`
- `trigger_phase`
- `trigger_condition`
- `text_template`
- `base_weight`
- `repeat_rule`
- `chain_next_codes`
- `options`

### 6.1 trigger_phase

首版固定阶段：

- `pre_kickoff`
- `first_half`
- `half_time`
- `second_half`
- `late_game`

### 6.2 trigger_condition

触发条件使用结构化条件集合，不写可执行脚本。

允许条件字段：

- `score_diff_range`
- `minute_bucket`
- `team_morale_range`
- `team_fitness_range`
- `has_yellow_risk_player`
- `has_core_player`
- `formation_tags`
- `tactic_tags`
- `event_history_tags`
- `player_status_tags`

### 6.3 repeat_rule

首版允许三种重复规则：

- `once_per_match`
- `once_per_phase`
- `repeatable_with_cooldown`

## 7. 事件执行流程

### 7.1 流程步骤

1. 比赛引擎进入某个事件阶段
2. 事件系统读取当前 `MatchContext`
3. 筛选满足 `trigger_phase` 和 `trigger_condition` 的候选模板
4. 根据 `base_weight` 和上下文修正计算触发权重
5. 选出当前要触发的事件
6. 返回事件文本和选项给 UI 层
7. 玩家选择某个选项
8. 事件系统解析 `EventEffect`
9. 比赛引擎把效果写回 `MatchContext`
10. 将触发记录写入事件历史

### 7.2 候选事件筛选原则

- 同一阶段不强制每次都触发事件
- 已触发过且被 `repeat_rule` 禁止的事件不可再次进入候选
- 明显不符合当前比分/状态的事件不进入候选
- 高优先级事件可以提升权重，但不能无条件覆盖全部事件

### 7.3 权重计算原则

首版权重计算公式采用线性修正：

- `final_weight = base_weight + context_bonus + tactic_bonus + player_bonus + chain_bonus`

约束：

- 权重最小值为 `0`
- 权重为 `0` 的事件不得触发
- 同类事件同时存在时，优先选择权重更高者

## 8. 事件效果模型

### 8.1 首版允许的效果类型

- `modify_team_morale`
- `modify_team_fitness`
- `modify_attack_bias`
- `modify_defense_bias`
- `modify_event_weight`
- `mark_player_priority`
- `swap_player`
- `add_context_tag`

### 8.2 效果执行约束

- 效果只修改比赛上下文，不直接生成终场比分
- 同一选项可以包含多个效果
- 效果执行顺序固定为：
  - 阵容变更
  - 球员状态变更
  - 团队修正变更
  - 后续事件权重变更
  - 上下文标签写入

### 8.3 连锁事件规则

- 首版只允许轻量连锁
- 一个事件最多挂接 2 个潜在后续事件 `code`
- 连锁只提升或降低后续权重，不直接强制触发

## 9. 事件文案与逻辑分离

必须遵守以下约束：

- 文本模板只负责展示，不承载业务逻辑
- 效果定义只描述结构化数据，不内嵌自然语言判断
- UI 只展示模板内容和选项，不解释比赛规则
- 测试时可直接绕过 UI 调用事件系统

## 10. 首版事件模板最低配额

首版建议最低模板数量：

- 临时战术安排：8 个
- 特殊换人事件：8 个
- 球员需求事件：6 个
- 裁判争议事件：6 个

最低总量：

- `28` 个事件模板

这样可以保证一届赛事中的事件重复感不会过强，同时仍保持首版可控。

## 11. 单元测试要求

事件系统单元测试至少覆盖：

- 条件筛选是否正确
- 重复规则是否正确
- 权重计算是否正确
- 事件选项效果是否正确写回上下文
- 连锁事件权重是否正确变化

建议单元测试用例：

- 用例 1：比分落后且进入 `late_game` 时，强攻类事件进入候选；领先保守局面不进入
- 用例 2：`once_per_match` 事件触发一次后，本场剩余阶段不可重复触发
- 用例 3：高风险战术模板下，激进事件的最终权重高于保守战术
- 用例 4：选择“换下体能不足边锋”后，阵容和体能修正立即更新
- 用例 5：选择“向裁判施压”后，裁判争议类后续事件权重发生可预期变化

## 12. 集成测试要求

事件系统集成测试至少覆盖：

- 事件仓储读取模板后，比赛引擎可正确消费
- UI 层可展示事件文本和选项，并把选择回传给事件系统
- 事件效果会影响赛果计算输入
- 事件日志能正确写入存档快照

建议集成测试用例：

- 用例 1：`EventRepository -> Event System` 联动时，读取模板后能产生合法候选事件
- 用例 2：`Match Engine -> Event System -> Match Engine` 往返后，比赛上下文修正值发生变化
- 用例 3：玩家在 UI 选择事件选项后，赛后快照中能看到对应事件日志
- 用例 4：同样阵容下，选择不同事件选项会导致结果修正摘要不同

## 13. 非目标

首版事件系统当前不实现：

- 自然语言生成事件文本
- 动态脚本执行
- 复杂多分支剧情树
- 多轮长期人格养成
