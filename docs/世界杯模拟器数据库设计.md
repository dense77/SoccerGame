# 世界杯模拟器数据库设计

## 1. 文档目标

本文档定义《世界杯模拟器》首版数据库结构，作为后续 SQLite 初始化、仓储层实现、测试设计和数据录入的统一基线。

首版数据库目标：

- 承载 2026 世界杯真实球队和球员基础数据
- 承载赛程模板、事件模板和战术模板
- 承载玩家存档、比赛结果和跨场状态
- 支撑前端单仓、浏览器内嵌 SQLite 的读取与持久化

## 2. 设计原则

- 静态基础数据和运行态存档数据分层存储
- 表结构优先服务玩法闭环，不追求一次性设计过度复杂
- 业务规则不直接写入表结构，规则由模块层解释
- 尽量通过稳定主键和外键关系保持后续可扩展性
- 数据访问统一走仓储层，上层 UI 不直接触达 SQL

## 3. 数据分层

### 3.1 静态基础数据

用于承载随版本发布的内容：

- 球队
- 球员
- 阵型
- 战术模板
- 赛程模板
- 事件模板

### 3.2 运行态存档数据

用于承载玩家游戏进度：

- 存档入口
- 当前赛事进度
- 已完成比赛
- 当前积分榜快照
- 球员状态快照
- 阵容与战术保存
- 事件历史摘要

## 4. 逻辑表清单

首版固定使用以下表：

- `teams`
- `players`
- `formations`
- `tactic_profiles`
- `team_tactic_profiles`
- `tournament_groups`
- `match_fixtures`
- `event_templates`
- `event_options`
- `save_slots`
- `save_team_states`
- `save_player_states`
- `save_match_setups`
- `match_snapshots`
- `match_event_logs`

## 5. 表结构定义

### 5.1 teams

用途：

- 存储参赛国家队基础信息和风格标签

字段：

- `id` TEXT PRIMARY KEY
- `name` TEXT NOT NULL
- `short_name` TEXT NOT NULL
- `fifa_code` TEXT NOT NULL UNIQUE
- `group_code` TEXT NOT NULL
- `default_formation_id` TEXT NOT NULL
- `style_tags_json` TEXT NOT NULL
- `overall_rating` INTEGER NOT NULL
- `goalkeeping_rating` INTEGER NOT NULL
- `defense_rating` INTEGER NOT NULL
- `midfield_rating` INTEGER NOT NULL
- `attack_rating` INTEGER NOT NULL
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL

约束：

- `overall_rating`、分路评分取值范围统一为 `1-100`
- `style_tags_json` 使用 JSON 数组字符串，保存如 `["control","wing_play"]`

### 5.2 players

用途：

- 存储球队正式名单球员资料和基础能力

字段：

- `id` TEXT PRIMARY KEY
- `team_id` TEXT NOT NULL
- `name` TEXT NOT NULL
- `shirt_number` INTEGER NOT NULL
- `primary_position` TEXT NOT NULL
- `secondary_positions_json` TEXT NOT NULL
- `overall_rating` INTEGER NOT NULL
- `attack_rating` INTEGER NOT NULL
- `defense_rating` INTEGER NOT NULL
- `stamina_rating` INTEGER NOT NULL
- `mentality_rating` INTEGER NOT NULL
- `position_fit_rating` INTEGER NOT NULL
- `is_core_player` INTEGER NOT NULL DEFAULT 0
- `status_tag` TEXT NOT NULL DEFAULT 'available'
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL

外键：

- `team_id` REFERENCES `teams(id)`

约束：

- `shirt_number` 在同一支球队内唯一
- `is_core_player` 使用 `0/1`
- `status_tag` 首版允许值：`available`、`good_form`、`fatigued`、`suspended`、`injured`

### 5.3 formations

用途：

- 存储阵型定义和位置槽位结构

字段：

- `id` TEXT PRIMARY KEY
- `name` TEXT NOT NULL UNIQUE
- `slot_layout_json` TEXT NOT NULL
- `tactic_tags_json` TEXT NOT NULL
- `recommended_usage` TEXT NOT NULL
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL

说明：

- `slot_layout_json` 保存阵型位置槽结构，例如 `["GK","LB","CB","CB","RB","CM","CM","LW","AM","RW","ST"]`

### 5.4 tactic_profiles

用途：

- 存储通用战术模板

字段：

- `id` TEXT PRIMARY KEY
- `name` TEXT NOT NULL UNIQUE
- `attack_bias` INTEGER NOT NULL
- `defensive_line` INTEGER NOT NULL
- `tempo` INTEGER NOT NULL
- `risk_appetite` INTEGER NOT NULL
- `event_weight_mod_json` TEXT NOT NULL
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL

约束：

- 数值字段统一采用 `-2` 到 `2` 的离散档位
- `event_weight_mod_json` 保存对事件类别或标签的权重修正

### 5.5 team_tactic_profiles

用途：

- 建立球队与默认战术模板的映射关系

字段：

- `id` TEXT PRIMARY KEY
- `team_id` TEXT NOT NULL
- `tactic_profile_id` TEXT NOT NULL
- `is_default` INTEGER NOT NULL DEFAULT 0
- `priority_order` INTEGER NOT NULL

外键：

- `team_id` REFERENCES `teams(id)`
- `tactic_profile_id` REFERENCES `tactic_profiles(id)`

### 5.6 tournament_groups

用途：

- 存储小组赛分组信息

字段：

- `id` TEXT PRIMARY KEY
- `group_code` TEXT NOT NULL UNIQUE
- `team_ids_json` TEXT NOT NULL
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL

说明：

- `team_ids_json` 保存该小组 4 支球队的 ID 列表

### 5.7 match_fixtures

用途：

- 存储小组赛和淘汰赛赛程模板

字段：

- `id` TEXT PRIMARY KEY
- `stage` TEXT NOT NULL
- `round_code` TEXT NOT NULL
- `group_code` TEXT
- `home_team_id` TEXT
- `away_team_id` TEXT
- `fixture_order` INTEGER NOT NULL
- `knockout_slot` TEXT
- `depends_on_json` TEXT NOT NULL DEFAULT '[]'
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL

说明：

- 小组赛阶段可直接存放球队 ID
- 淘汰赛阶段可通过 `depends_on_json` 记录上游胜者来源，如 `["A1","B2"]`

### 5.8 event_templates

用途：

- 存储关键事件模板定义

字段：

- `id` TEXT PRIMARY KEY
- `category` TEXT NOT NULL
- `code` TEXT NOT NULL UNIQUE
- `title` TEXT NOT NULL
- `trigger_phase` TEXT NOT NULL
- `trigger_condition_json` TEXT NOT NULL
- `text_template` TEXT NOT NULL
- `base_weight` INTEGER NOT NULL
- `repeat_rule` TEXT NOT NULL
- `chain_next_codes_json` TEXT NOT NULL DEFAULT '[]'
- `effect_summary_json` TEXT NOT NULL
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL

说明：

- `category` 固定为 `tactic_adjustment`、`substitution`、`player_demand`、`referee_conflict`

### 5.9 event_options

用途：

- 存储事件可选项和效果

字段：

- `id` TEXT PRIMARY KEY
- `event_template_id` TEXT NOT NULL
- `option_key` TEXT NOT NULL
- `label` TEXT NOT NULL
- `effect_json` TEXT NOT NULL
- `sort_order` INTEGER NOT NULL

外键：

- `event_template_id` REFERENCES `event_templates(id)`

### 5.10 save_slots

用途：

- 存储玩家存档主记录

字段：

- `id` TEXT PRIMARY KEY
- `selected_team_id` TEXT NOT NULL
- `current_stage` TEXT NOT NULL
- `current_round_code` TEXT NOT NULL
- `status` TEXT NOT NULL
- `created_at` TEXT NOT NULL
- `updated_at` TEXT NOT NULL

外键：

- `selected_team_id` REFERENCES `teams(id)`

说明：

- `status` 允许值：`active`、`eliminated`、`champion`

### 5.11 save_team_states

用途：

- 存储某个存档下各支球队的赛事状态快照

字段：

- `id` TEXT PRIMARY KEY
- `save_slot_id` TEXT NOT NULL
- `team_id` TEXT NOT NULL
- `group_points` INTEGER NOT NULL DEFAULT 0
- `goal_for` INTEGER NOT NULL DEFAULT 0
- `goal_against` INTEGER NOT NULL DEFAULT 0
- `goal_diff` INTEGER NOT NULL DEFAULT 0
- `wins` INTEGER NOT NULL DEFAULT 0
- `draws` INTEGER NOT NULL DEFAULT 0
- `losses` INTEGER NOT NULL DEFAULT 0
- `is_eliminated` INTEGER NOT NULL DEFAULT 0
- `current_morale_mod` INTEGER NOT NULL DEFAULT 0
- `current_fitness_mod` INTEGER NOT NULL DEFAULT 0
- `updated_at` TEXT NOT NULL

外键：

- `save_slot_id` REFERENCES `save_slots(id)`
- `team_id` REFERENCES `teams(id)`

### 5.12 save_player_states

用途：

- 存储某个存档下球员跨场状态

字段：

- `id` TEXT PRIMARY KEY
- `save_slot_id` TEXT NOT NULL
- `player_id` TEXT NOT NULL
- `status_tag` TEXT NOT NULL
- `fitness_value` INTEGER NOT NULL
- `morale_value` INTEGER NOT NULL
- `yellow_card_count` INTEGER NOT NULL DEFAULT 0
- `injury_flag` INTEGER NOT NULL DEFAULT 0
- `suspension_flag` INTEGER NOT NULL DEFAULT 0
- `last_match_minutes` INTEGER NOT NULL DEFAULT 0
- `updated_at` TEXT NOT NULL

外键：

- `save_slot_id` REFERENCES `save_slots(id)`
- `player_id` REFERENCES `players(id)`

### 5.13 save_match_setups

用途：

- 存储某个存档对某场比赛的赛前配置

字段：

- `id` TEXT PRIMARY KEY
- `save_slot_id` TEXT NOT NULL
- `fixture_id` TEXT NOT NULL
- `team_id` TEXT NOT NULL
- `formation_id` TEXT NOT NULL
- `tactic_profile_id` TEXT NOT NULL
- `starting_player_ids_json` TEXT NOT NULL
- `bench_player_ids_json` TEXT NOT NULL
- `updated_at` TEXT NOT NULL

外键：

- `save_slot_id` REFERENCES `save_slots(id)`
- `fixture_id` REFERENCES `match_fixtures(id)`
- `team_id` REFERENCES `teams(id)`
- `formation_id` REFERENCES `formations(id)`
- `tactic_profile_id` REFERENCES `tactic_profiles(id)`

### 5.14 match_snapshots

用途：

- 存储已完成比赛的赛后快照

字段：

- `id` TEXT PRIMARY KEY
- `save_slot_id` TEXT NOT NULL
- `fixture_id` TEXT NOT NULL
- `stage` TEXT NOT NULL
- `home_team_id` TEXT NOT NULL
- `away_team_id` TEXT NOT NULL
- `home_score` INTEGER NOT NULL
- `away_score` INTEGER NOT NULL
- `result_summary_json` TEXT NOT NULL
- `applied_modifiers_json` TEXT NOT NULL
- `completed_at` TEXT NOT NULL

外键：

- `save_slot_id` REFERENCES `save_slots(id)`
- `fixture_id` REFERENCES `match_fixtures(id)`

### 5.15 match_event_logs

用途：

- 存储单场比赛中实际触发的事件记录

字段：

- `id` TEXT PRIMARY KEY
- `match_snapshot_id` TEXT NOT NULL
- `event_template_id` TEXT NOT NULL
- `option_id` TEXT NOT NULL
- `phase` TEXT NOT NULL
- `sequence_no` INTEGER NOT NULL
- `resolved_effect_json` TEXT NOT NULL
- `created_at` TEXT NOT NULL

外键：

- `match_snapshot_id` REFERENCES `match_snapshots(id)`
- `event_template_id` REFERENCES `event_templates(id)`
- `option_id` REFERENCES `event_options(id)`

## 6. 索引设计

首版至少建立以下索引：

- `players(team_id)`
- `teams(group_code)`
- `match_fixtures(stage, round_code, fixture_order)`
- `event_templates(category, trigger_phase)`
- `save_team_states(save_slot_id, team_id)`
- `save_player_states(save_slot_id, player_id)`
- `save_match_setups(save_slot_id, fixture_id)`
- `match_snapshots(save_slot_id, fixture_id)`
- `match_event_logs(match_snapshot_id, sequence_no)`

## 7. 仓储层接口约定

数据库访问统一经由仓储层，首版至少需要以下仓储接口族：

- `TeamRepository`
  - 获取全部参赛球队
  - 按 ID 获取球队
  - 获取球队球员名单
- `TournamentRepository`
  - 获取分组
  - 获取指定阶段赛程
  - 获取下一轮赛程模板
- `TacticRepository`
  - 获取阵型定义
  - 获取战术模板
  - 获取球队默认战术
- `EventRepository`
  - 按类别和阶段获取事件模板
  - 获取事件选项
- `SaveRepository`
  - 创建存档
  - 读取存档
  - 保存球队状态
  - 保存球员状态
  - 保存赛前配置
  - 保存比赛快照与事件日志

## 8. 初始化与迁移策略

首版初始化策略：

- 项目首次运行时执行 schema 初始化
- 若静态基础数据为空，则导入球队、球员、阵型、战术、赛程、事件模板
- 运行态存档表始终为空库创建

首版迁移策略：

- 使用显式数据库版本号
- 新增字段优先采用兼容默认值
- 不允许在没有迁移脚本的情况下直接破坏旧存档结构

## 9. 单元测试要求

数据库设计相关的单元测试至少覆盖：

- schema 初始化是否创建全部核心表
- 表约束是否生效
- 仓储查询是否返回稳定结构
- 重复初始化是否幂等
- 存档写入是否只影响对应 `save_slot`

建议单元测试用例：

- 用例 1：初始化后存在 `teams`、`players`、`match_fixtures`、`save_slots`、`match_snapshots`
- 用例 2：插入不存在的 `team_id` 到 `players` 时应失败
- 用例 3：同一球队重复插入相同 `shirt_number` 时应失败
- 用例 4：按球队 ID 查询球员时返回 26 人且结构完整
- 用例 5：创建两个存档后，更新其中一个的球员状态不会污染另一个

## 10. 集成测试要求

数据库设计相关的集成测试至少覆盖：

- 初始化基础数据后，赛事模块可以读取完整分组与赛程
- 选队后可以创建存档并初始化球队/球员状态
- 比赛结束后可以保存比赛快照和事件日志
- 重新加载存档后可以恢复当前轮次、球员状态和赛前阵容

建议集成测试用例：

- 用例 1：`TeamRepository + TournamentRepository` 联合读取时，能构建选队页与小组赛首轮赛程
- 用例 2：`SaveRepository + TeamRepository` 联合运行时，创建存档后能读取玩家球队及默认配置
- 用例 3：`Match Engine -> SaveRepository` 写入比赛结果后，`Tournament Module` 能继续推进积分榜
- 用例 4：`EventRepository -> Match Engine -> SaveRepository` 联动时，事件模板触发和赛后落盘结构一致

## 11. 非目标

本数据库设计当前不解决以下问题：

- 联网同步
- 多设备云存档
- 复杂统计查询报表
- 多赛季历史归档
