# 世界杯模拟器开发计划

## 1. 文档目的

本文档基于 [世界杯模拟器方案](E:/SoccerGame/docs/世界杯模拟器方案.md:1) 拆解首版开发路径，目标是把“产品方案”转换为“可执行的工程计划”。

适用范围：

- 首版单支国家队冲击 2026 世界杯冠军模式
- 前端单仓项目
- 浏览器内嵌 SQLite 数据方案
- 关键事件回合制比赛系统

本文档只定义开发顺序、阶段目标、测试方法和测试用例，不替代详细的数据库设计文档和事件模板文档。

## 2. 开发原则

- 先打通最小可玩闭环，再逐步丰富内容。
- 先稳定模块边界，再增加事件和数值复杂度。
- 先保证数据结构和赛程推进正确，再优化 UI 表现。
- 每个阶段都必须有可验证产物，避免一次性堆大功能。

## 3. 阶段总览

### 阶段 0：基础设计冻结

目标：

- 把后续实现要依赖的结构定义清楚，避免边写边改。

产出：

- 数据库设计文档
- 事件系统设计文档
- 比赛结算规则说明
- 工程目录约定

完成标准：

- 关键实体、表结构、模块职责、输入输出已固定
- 后续开发不再讨论“数据放哪里”“模块怎么拆”“事件长什么样”

### 阶段 1：数据层与基础工程搭建

目标：

- 让项目具备读取球队、球员、阵型、赛程和事件模板的能力。

产出：

- 浏览器 SQLite 接入
- 数据初始化机制
- 数据访问层接口
- 基础类型定义

完成标准：

- 可以从数据库读取至少 1 支球队和完整球员列表
- 可以读取赛程模板和事件模板
- UI 层不直接接触底层 SQL

### 阶段 2：赛事骨架与存档骨架

目标：

- 打通“选队 -> 进入小组赛 -> 推进赛程 -> 保存当前进度”的主流程。

产出：

- 选队流程
- 赛事状态管理
- 小组赛与淘汰赛赛程推进
- 存档入口和继续游戏能力

完成标准：

- 玩家能选择一支队伍创建存档
- 赛事能正确记录当前轮次、积分和晋级状态

### 阶段 3：球队管理与赛前准备

目标：

- 让玩家在每场比赛前完成阵容和战术设置。

产出：

- 球队名单页
- 首发/替补选择
- 阵型选择
- 战术倾向设置
- 出场合法性校验

完成标准：

- 能提交一套合法阵容进入比赛
- 球员状态会影响可选阵容和赛前提示

### 阶段 4：比赛引擎最小闭环

目标：

- 先实现不依赖复杂事件的基础赛果计算。

产出：

- 比赛上下文模型
- 基础实力分计算
- 阵型/战术修正分计算
- 比分生成与赛后结果输出

完成标准：

- 同一套输入能稳定生成可解释结果
- 强队对弱队有更高胜率，但不是绝对碾压

### 阶段 5：关键事件系统

目标：

- 把比赛从“纯结算”升级成“有决策参与”的文字模拟。

产出：

- 事件模板加载机制
- 事件触发器
- 事件选项与效果执行
- 事件历史记录
- 四类首版事件落地

完成标准：

- 比赛中能按上下文触发多类事件
- 玩家选择能改变比赛修正值或赛后状态

### 阶段 6：赛后反馈与跨场状态延续

目标：

- 让比赛结果真正影响下一场，形成连续性。

产出：

- 球员体能变化
- 情绪变化
- 停赛/伤病状态
- 赛后摘要和关键反馈

完成标准：

- 上一场的决策和结果会在下一场显著体现

### 阶段 7：UI 串联与首版验收

目标：

- 完成从开局到夺冠的完整可玩版本。

产出：

- 页面流程串联
- 错误态和空态处理
- 基础文案补齐
- 首版验收测试

完成标准：

- 玩家能完成一整届世界杯流程
- 所有关键路径可复现、可验证、可回归

## 4. 详细开发步骤

### 步骤 1：补齐设计基线

实现内容：

- 新增数据库设计文档，明确表结构、主键、外键、索引和字段含义。
- 新增事件系统设计文档，明确事件模板格式、触发条件格式、效果格式。
- 新增比赛规则文档，明确能力权重、阵型克制修正和比分输出规则。
- 明确工程目录结构，至少拆出 `data`、`domain`、`modules`、`ui`、`types`。

测试方法：

- 文档审查
- 结构一致性检查

测试用例：

- 用例 1：`Team`、`Player`、`MatchEvent`、`SaveSlot` 是否都能映射到数据库结构。
- 用例 2：事件模板是否能表达四类首版事件。
- 用例 3：模块划分是否满足“UI 不直接计算赛果、数据层不内嵌规则”。

### 步骤 2：实现 SQLite 基础设施

实现内容：

- 集成浏览器 SQLite 方案。
- 建立数据库初始化流程。
- 写入基础表结构和种子数据装载机制。
- 建立仓储层接口，封装球队、球员、赛程、事件、存档查询。

测试方法：

- 启动时数据库初始化测试
- 仓储接口单元测试
- 数据读取手工验证

测试用例：

- 用例 1：首次启动时能创建数据库并写入基础数据。
- 用例 2：重复启动不会重复插入相同球队和球员数据。
- 用例 3：按球队 ID 能读取该队完整 26 人名单。
- 用例 4：能读取指定小组全部球队和赛程信息。
- 用例 5：UI 层通过仓储接口拿到数据，不直接调用 SQL。

### 步骤 3：建立核心类型与模块骨架

实现内容：

- 定义 `Team`、`Player`、`Formation`、`Match`、`MatchContext`、`MatchEvent`、`SaveSlot` 等核心类型。
- 为 `Tournament Module`、`Team Management Module`、`Match Engine Module`、`Event System Module` 建立独立入口。
- 明确模块间只通过输入输出对象交互。

测试方法：

- 类型检查
- 模块边界静态检查
- 小范围假数据联调

测试用例：

- 用例 1：比赛引擎输入不依赖 React 组件实例。
- 用例 2：事件系统可以脱离 UI 独立运行。
- 用例 3：赛事模块可以只依赖比赛结果对象推进赛程。

### 步骤 4：实现选队和存档创建

实现内容：

- 增加球队选择页。
- 创建新存档并绑定玩家所选球队。
- 初始化该队赛程上下文、积分状态和球员初始状态。
- 支持读取已有存档继续游戏。

测试方法：

- 手工流程测试
- 存档创建与读取测试

测试用例：

- 用例 1：选择任意球队后能创建新存档并进入赛事主界面。
- 用例 2：已有存档再次进入时能恢复当前轮次和已完成比赛结果。
- 用例 3：不同存档选择不同球队时，彼此数据互不污染。

### 步骤 5：实现赛事推进骨架

实现内容：

- 落地小组赛赛程推进。
- 实现积分、净胜球、胜平负统计。
- 实现小组晋级判断。
- 实现淘汰赛对阵推进到冠军结算。

测试方法：

- 纯逻辑单元测试
- 赛程推进集成测试
- 边界值测试

测试用例：

- 用例 1：胜利计 3 分、平局计 1 分、失败计 0 分。
- 用例 2：同分情况下按净胜球排序。
- 用例 3：小组赛 3 轮结束后能正确选出晋级队。
- 用例 4：淘汰赛胜者进入下一轮，败者被淘汰。
- 用例 5：决赛结束后能生成冠军结果并结束存档流程。

### 步骤 6：实现球队管理和赛前配置

实现内容：

- 球队名单展示。
- 球员状态展示。
- 首发与替补设置。
- 阵型切换。
- 战术倾向设置。
- 阵容合法性校验和异常提示。

测试方法：

- UI 交互测试
- 阵容校验逻辑测试
- 状态联动测试

测试用例：

- 用例 1：必须选择合法人数的首发和替补才能进入比赛。
- 用例 2：伤病和停赛球员不可进入首发。
- 用例 3：非主位置球员进入首发会带来适配度警告或惩罚提示。
- 用例 4：切换阵型后，位置槽和球员可用性同步刷新。
- 用例 5：保存阵容后，下次进入同一比赛前页面仍能恢复设置。

### 步骤 7：实现基础比赛引擎

实现内容：

- 初始化比赛上下文。
- 计算首发综合能力和状态权重。
- 计算阵型克制和战术相性。
- 生成基础赛果和比分区间。
- 输出赛后结果对象。

测试方法：

- 纯逻辑单元测试
- 固定输入回归测试
- 权重调试测试

测试用例：

- 用例 1：同阵容、同状态、同参数下，多次运行结果应在预期范围内波动，不出现异常极值。
- 用例 2：强队完整主力对弱队替补阵容时，强队胜率应明显更高。
- 用例 3：主力缺阵、疲劳或情绪异常会降低基础实力分。
- 用例 4：阵型克制变化会影响比赛修正分，但不会完全颠覆极端实力差。
- 用例 5：赛后结果对象包含比分、修正值摘要和后续上下文。

### 步骤 8：实现事件系统最小闭环

实现内容：

- 事件模板读取。
- 事件触发条件判定。
- 事件选项呈现。
- 事件效果执行。
- 事件链记录。

测试方法：

- 事件模板单元测试
- 条件触发测试
- 事件效果集成测试

测试用例：

- 用例 1：比分落后时更容易触发临时强攻类事件。
- 用例 2：主力黄牌或体能不足时可触发特殊换人事件。
- 用例 3：核心球员状态较高时可触发球员需求事件。
- 用例 4：裁判争议事件的不同选择会产生不同风险收益。
- 用例 5：某事件选择后，后续关联事件的触发权重发生变化。

### 步骤 9：实现跨场状态与赛后反馈

实现内容：

- 赛后更新球员体能、情绪、停赛和伤病。
- 生成赛后摘要和关键决策回顾。
- 将结果落盘到存档。
- 在下一场赛前读取并展示这些变化。

测试方法：

- 状态迁移测试
- 存档快照测试
- 手工连续比赛验证

测试用例：

- 用例 1：高强度比赛后主力体能下降，下一场赛前可见。
- 用例 2：裁判争议或球员需求处理不当会影响后续情绪状态。
- 用例 3：停赛球员在下一场被正确标记为不可出场。
- 用例 4：读取存档后，上一场赛后的状态变化不会丢失。

### 步骤 10：完成前端串联和首版验收

实现内容：

- 串联开局、赛前、比赛、赛后、下一轮页面流程。
- 补齐加载态、空态、异常态。
- 增加基础战报和结果说明。
- 完成首版回归测试清单。

测试方法：

- 全流程手工冒烟测试
- 回归测试
- 关键路径验收测试

测试用例：

- 用例 1：从新建存档开始，完整推进 1 场比赛无阻塞。
- 用例 2：完整推进小组赛 3 场，积分榜和晋级结果正确。
- 用例 3：完整推进淘汰赛直到冠军，流程不丢状态。
- 用例 4：中途关闭页面再恢复存档，赛事进度与阵容状态一致。
- 用例 5：至少四类事件在整届赛事中都有合理触发机会。

## 5. 测试策略

### 5.1 测试层次

- 单元测试：覆盖赛事推进、赛果计算、事件触发、阵容合法性、状态迁移。
- 集成测试：覆盖数据层到模块层的联调，如“读取球队 -> 排阵 -> 结算 -> 推进赛事”。
- 手工流程测试：覆盖真实操作体验和页面串联。
- 回归测试：每次调整事件权重、阵型修正或存档结构后重新执行关键用例。

### 5.2 测试顺序

建议按以下顺序测试：

1. 先测纯逻辑模块  
   赛事推进、比赛引擎、事件系统。
2. 再测数据和存档  
   初始化、读取、恢复。
3. 最后测 UI 全链路  
   开局、赛前、比赛、赛后、继续游戏。

### 5.3 首版重点风险

- SQLite 在浏览器内的初始化、持久化和版本升级策略不稳定。
- 事件系统如果模板结构设计不清晰，后续扩展会非常痛苦。
- 赛果权重如果过早写死在 UI 或页面状态里，后面难以替换。
- 球员状态和赛事进度如果没有统一快照模型，存档恢复容易错乱。

因此首版测试必须优先验证：

- 数据层是否稳定
- 模块边界是否清晰
- 存档恢复是否可信
- 赛果是否可解释

## 6. 建议的交付顺序

建议严格按以下顺序推进，不要跳步骤并行堆功能：

1. 数据库设计文档
2. 事件系统设计文档
3. SQLite 基础设施
4. 核心类型与模块骨架
5. 选队与存档
6. 赛事推进
7. 球队管理
8. 基础比赛引擎
9. 事件系统
10. 赛后状态延续
11. UI 串联与验收

## 7. 完成定义

首版开发完成需同时满足以下条件：

- 能从选队开始完成一整届世界杯流程
- 阵容、战术、事件选择都会影响比赛结果
- 赛后状态会影响下一场
- 存档可恢复
- 模块边界保持解耦
- 数据和规则调整不需要重写页面结构
