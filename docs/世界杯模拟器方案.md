# 世界杯模拟器方案

## 1. 文档定位

本文档用于沉淀《世界杯模拟器》首版的产品方案与系统设计，作为后续实现、拆分任务、补充数据和调整数值的统一基准。

当前项目已有的 `Dexie/IndexedDB` 相关文档视为早期技术选型背景，不再作为首版默认落地方案。首版默认数据方案改为 `浏览器内嵌 SQLite`，目标是承载真实的 2026 世界杯球队与球员数据，并为后续玩法升级保留清晰的模块边界。

## 2. 项目概述

### 2.1 游戏主题

- 游戏名称：世界杯模拟器
- 类型：足球文字模拟游戏
- 题材：2026 世界杯国家队征程
- 玩家身份：国家队主教练

### 2.2 核心体验

玩家执教一支真实的 2026 世界杯参赛国家队，从小组赛一路推进到淘汰赛和决赛。游戏不提供实时操作，而是通过赛前阵容安排和比赛中的关键事件决策，让玩家体验“主教练在重大节点做判断”的乐趣。

核心体验目标：

- 让玩家感受到世界杯赛程推进的紧张感
- 让阵容选择和临场决策对结果产生可解释的影响
- 让每场比赛既有基础实力差异，也有事件驱动的不确定性
- 让系统结构保持解耦，便于后续扩展成更复杂的足球模拟器

### 2.3 首版范围

首版只覆盖“单支国家队冲击本届世界杯冠军”的完整闭环，不包含以下内容：

- 多赛季执教
- 转会系统
- 球员成长养成
- 俱乐部玩法
- 经济系统
- 董事会系统
- 实时操作比赛

## 3. 核心玩法循环

游戏的主循环固定为以下 5 个阶段：

1. 选择球队  
   玩家从 2026 世界杯参赛国家队中选择一支作为当前存档执教对象。
2. 赛前备战  
   玩家查看对手、积分榜、球员状态和可用名单，安排首发、替补、阵型与战术倾向。
3. 比赛过程  
   比赛采用关键事件回合制推进。系统在关键节点抛出事件，玩家做选择并即时影响比赛走势。
4. 赛后结算  
   系统给出比分、关键事件回顾、球员状态变化、停赛伤病变化和积分榜更新。
5. 下一场准备  
   进入下一轮赛事，继续调整阵容并处理新的球队内部问题。

这一闭环要支持：

- 小组赛 3 场完整推进
- 晋级判断、积分榜更新和净胜球比较
- 淘汰赛晋级到下一轮
- 直至冠军产生

## 4. 单局结构

### 4.1 存档主线

单个存档只允许玩家执教一支国家队，不支持在赛事途中切换队伍。

单局结构：

- 选择国家队
- 载入该队 26 人正式名单与默认阵型建议
- 进入小组赛阶段
- 逐场推进并记录成绩、球员状态和事件结果
- 根据赛果进入淘汰赛或出局结算

### 4.2 比赛外操作

比赛外界面至少支持以下操作：

- 查看球队名单
- 查看球员位置、能力、状态、体能、情绪和可出场情况
- 设置首发与替补
- 选择阵型
- 选择战术倾向
- 查看分组、积分榜和赛程
- 查看上一场事件记录与关键反馈

### 4.3 比赛内交互节奏

首版采用“关键事件回合制”，不按逐分钟实时操作，不要求玩家频繁点选。

单场比赛流程建议为：

1. 赛前对阵摘要
2. 开场状态结算
3. 事件触发与选择
4. 中场摘要
5. 下半场事件触发与选择
6. 终场结算

比赛推进的本质不是时间流逝模拟，而是“按关键节点抽取事件并累积结果修正值”。

## 5. 单场比赛系统

### 5.1 设计目标

单场比赛系统应满足以下要求：

- 能体现双方纸面实力差异
- 能体现阵型和战术的克制关系
- 能让玩家的事件选择产生清晰反馈
- 能保持结果可解释，避免黑箱
- 能和 UI、数据存储、事件文案解耦

### 5.2 比赛输入

比赛引擎在开场时至少接收以下输入：

- 我方球队基础数据
- 对方球队基础数据
- 我方首发、替补、阵型、战术倾向
- 对方默认阵容与战术模板
- 我方球员状态、体能、情绪、停赛伤病信息
- 对方球员状态模板
- 当前赛事阶段信息
- 事件池与触发条件

### 5.3 比赛输出

比赛结束后至少输出：

- 最终比分
- 进球归属或关键贡献摘要
- 触发事件列表
- 事件选择结果
- 球员状态变化
- 停赛、伤病、情绪变化
- 赛后战报摘要
- 可用于下一场比赛的上下文数据

### 5.4 事件类别

首版事件池限定为 4 类：

1. 临时战术安排  
   例如压上进攻、回收防线、重点盯防、边路提速。
2. 特殊换人事件  
   例如主力受伤风险升高、边锋状态火热、后腰被黄牌压制。
3. 球员需求事件  
   例如核心球员要求增加球权、替补希望登场、队长对战术安排提出异议。
4. 裁判争议事件  
   例如是否激烈抗议、是否克制情绪、是否向第四官员施压。

每类事件都必须有统一结构：

- `触发条件`
- `出现时机`
- `事件文本`
- `可选项`
- `即时效果`
- `持续效果`
- `后续事件关联`

### 5.5 事件触发原则

事件触发至少由以下因素共同决定：

- 当前比分
- 比赛阶段
- 双方实力差距
- 阵型对抗关系
- 球员状态和情绪
- 赛前战术设置
- 已经发生过的事件

事件系统必须支持以下能力：

- 同一模板在不同比赛上下文下触发不同文案
- 一个事件的选择影响后续事件权重
- 部分事件只在特定球员或特定阵型下触发

### 5.6 赛果计算规则

首版赛果计算由两层构成：

#### 1. 基础实力分

基础实力分由以下因子加权形成：

- 首发 11 人综合能力
- 位置适配度
- 球员当前状态
- 体能
- 情绪稳定度
- 阵容完整度

#### 2. 对抗修正分

对抗修正分由以下因子叠加形成：

- 阵型克制关系
- 战术倾向相性
- 关键事件选择结果
- 个别球员的临场表现加成或减成

#### 3. 结果生成原则

- 基础实力分决定双方的基础胜率区间
- 对抗修正分负责拉开或缩小比赛差距
- 关键事件应能改变比赛走势，但不应彻底抹平极端实力差
- 输出结果时要能追溯“为什么赢、为什么输”

### 5.7 阵型克制原则

首版只做轻量阵型克制，不做复杂战术树。

建议以“阵型标签”而非复杂空间模型处理克制关系，例如：

- 边路强势
- 中路压制
- 高位逼抢
- 防反稳健
- 控球组织

阵型克制只提供修正值，不直接决定胜负。这样可以在后续升级为更复杂的战术对抗模型时复用现有接口。

## 6. 数据设计

### 6.1 数据口径

首版球队和球员数据以 `2026 世界杯官方最终名单` 为目标口径。

如果实际录入阶段存在个别字段缺失，允许用占位值补齐，但不得改变整体结构，也不得把预测名单和正式名单混用成不清晰的状态。

### 6.2 核心实体

首版至少定义以下核心实体：

- `Team`
- `Player`
- `Formation`
- `TacticProfile`
- `Match`
- `MatchContext`
- `MatchEvent`
- `EventOption`
- `EventEffect`
- `TournamentStanding`
- `SaveSlot`

### 6.3 Team

`Team` 至少包含：

- 唯一标识
- 国家名称
- 国家缩写
- 所属小组
- 默认阵型
- 整体风格标签
- 球队综合评分
- 守门强度、后防强度、中场强度、进攻强度

### 6.4 Player

`Player` 至少包含：

- 唯一标识
- 所属球队标识
- 姓名
- 球衣号码
- 主位置
- 可兼容位置列表
- 综合能力
- 进攻倾向
- 防守倾向
- 体能
- 心态稳定
- 位置适配
- 初始状态
- 是否核心球员

首版不做细颗粒度属性爆炸，不引入几十个能力字段。以“够支撑结果计算”为原则。

### 6.5 Formation

`Formation` 至少包含：

- 阵型标识
- 阵型名称
- 位置槽位结构
- 战术标签
- 推荐使用条件

### 6.6 TacticProfile

`TacticProfile` 至少包含：

- 战术标识
- 战术名称
- 进攻倾向
- 防线高低
- 节奏快慢
- 风险偏好
- 对事件权重的影响项

### 6.7 Match

`Match` 至少包含：

- 唯一标识
- 赛事阶段
- 轮次
- 对阵双方
- 开球顺序或展示顺序
- 最终比分
- 是否完成

### 6.8 MatchContext

`MatchContext` 用于承载运行态信息，至少包含：

- 当前比赛阶段
- 当前比分
- 已触发事件记录
- 双方士气修正
- 双方体能修正
- 阵型克制修正
- 战术修正
- 玩家选择历史

### 6.9 MatchEvent

`MatchEvent` 至少包含：

- 事件标识
- 事件类别
- 触发条件
- 文本模板
- 可选项集合
- 权重
- 可重复触发规则
- 连锁事件标识

### 6.10 SaveSlot

`SaveSlot` 至少包含：

- 存档标识
- 玩家所选球队
- 当前赛事阶段
- 当前赛程进度
- 当前积分榜快照
- 球员状态快照
- 已完成比赛结果
- 事件历史摘要

## 7. SQLite 数据方案

### 7.1 首版默认存储策略

首版默认采用 `浏览器内嵌 SQLite`。其职责分为两层：

- 静态基础数据层：球队、球员、阵型、事件模板、赛程模板
- 运行存档层：存档、已完成比赛、球员状态快照、积分榜快照

这一路线的目标是：

- 使用统一的数据模型承载真实世界杯数据
- 让静态数据与运行存档具备一致的数据访问方式
- 后续若迁移到本地客户端或轻量后端时，能最大化复用结构

### 7.2 表职责建议

首版至少规划以下数据表：

- `teams`  
  存储参赛国家队基础信息和风格标签。
- `players`  
  存储球员名单、位置和能力数据。
- `formations`  
  存储可选阵型和位置槽结构。
- `team_tactics`  
  存储球队默认战术模板或可用倾向配置。
- `match_fixtures`  
  存储小组赛和淘汰赛赛程模板。
- `event_templates`  
  存储关键事件模板、条件与效果定义。
- `save_slots`  
  存储存档入口数据。
- `match_snapshots`  
  存储单场比赛结果、事件摘要和状态变化。

### 7.3 数据访问原则

数据访问层必须满足以下约束：

- 上层业务不直接拼装底层 SQL 细节
- 业务层通过统一仓储接口读取球队、球员、赛程和事件
- 事件模板与事件执行结果分离
- 存档快照与静态基础数据分离

### 7.4 与旧方案的关系

现有文档中的 `Dexie/IndexedDB` 描述保留为历史选型背景，不作为首版默认实现路径。

如果未来因为平台限制导致浏览器 SQLite 方案不合适，可将 `Data Access Module` 适配到其他本地存储实现，但上层玩法和比赛模块不应因此重写。

## 8. 模块解耦设计

### 8.1 Tournament Module

职责：

- 维护小组赛和淘汰赛进程
- 计算积分、净胜球和晋级结果
- 在赛后推进到下一轮

不负责：

- 单场事件决策
- UI 展示
- 原始数据库查询细节

### 8.2 Team Management Module

职责：

- 管理球队名单、首发、替补、阵型、战术配置
- 校验出场阵容的基本合法性
- 提供赛前数据给比赛引擎

不负责：

- 计算最终赛果
- 决定赛事晋级关系

### 8.3 Match Engine Module

职责：

- 初始化单场比赛上下文
- 根据输入生成关键事件流程
- 计算对抗修正分和最终结果
- 输出赛后快照

不负责：

- 事件文案的展示层实现
- 数据库存取

### 8.4 Event System Module

职责：

- 管理事件模板
- 判断事件能否触发
- 提供选项和效果
- 记录事件链影响

不负责：

- 直接渲染界面
- 直接维护赛事积分榜

### 8.5 Data Access Module

职责：

- 封装 SQLite 读写
- 提供球队、球员、赛程、事件和存档查询接口
- 屏蔽底层表结构变动带来的影响

不负责：

- 业务规则计算
- 赛果逻辑

### 8.6 Save/Progress Module

职责：

- 维护当前存档进度
- 记录已完成比赛、球员状态和事件摘要
- 支撑继续游戏与读取历史结果

不负责：

- 直接触发赛事逻辑

### 8.7 UI Orchestration Layer

职责：

- 组织页面流程
- 调用模块接口
- 展示比赛、球队、赛程和事件结果

不负责：

- 承载比赛规则
- 直接编写数据层逻辑

### 8.8 解耦约束

实现时必须遵守以下边界：

- UI 不直接计算赛果
- 比赛引擎不直接依赖 React 组件
- 事件文本与事件效果分离
- 数据访问层不内嵌玩法规则
- 赛事推进与单场比赛结算分离
- 赛果算法可替换，但对外输入输出结构尽量稳定

## 9. 首版系统行为约束

### 9.1 球员状态

首版球员状态至少覆盖：

- 可出场
- 轻度疲劳
- 良好状态
- 情绪波动
- 停赛不可出场
- 伤病不可出场

状态的作用范围：

- 影响基础实力分
- 影响事件触发概率
- 影响赛前排阵可选性

### 9.2 替补与换人

首版换人不做精细规则仿真，但要满足：

- 替补席来自赛前名单
- 换人事件可以改变实力修正值
- 特定球员上场可触发特定事件或加成

### 9.3 裁判与争议事件

裁判事件的核心价值是制造风险与收益权衡，不追求完整规则模拟。

例如：

- 强烈抗议可能带来短时士气提升，也可能导致教练或球员情绪失控
- 选择克制可能避免风险，但放弃部分施压收益

### 9.4 球员需求事件

球员需求事件的作用不是做深度人格系统，而是提供“主教练管理更衣室”的最小体验。

首版只需要支持：

- 核心球员表达战术偏好
- 替补表达出场诉求
- 队长表达团队稳定意见

## 10. 验收标准

当后续进入开发时，首版应至少达到以下验收标准：

- 玩家可以选择一支真实的 2026 世界杯国家队开始游戏
- 游戏可以完成从小组赛到淘汰赛直至冠军结算的完整流程
- 赛前可以设置首发、替补、阵型和战术倾向
- 单场比赛可以触发多类关键事件
- 不同事件选择会改变比赛修正值或赛后状态
- 积分榜、晋级关系和淘汰赛推进逻辑正确
- 球员状态会跨比赛保留并影响下一场决策
- 替换事件模板或调整赛果权重时，不需要重写 UI 层
- 后续升级更复杂的比赛模型时，不需要推翻模块边界

## 11. 后续升级路线

首版完成后，后续升级建议按以下顺序展开：

1. 丰富事件池  
   增加媒体舆论、队内冲突、点球大战心理博弈、天气场地等事件。
2. 细化数值系统  
   扩充球员能力模型和战术相性权重。
3. 增强比赛表现层  
   提供更完整的比赛文字播报和战报摘要。
4. 强化存档系统  
   支持多存档、历史战绩查询和赛事回放摘要。
5. 扩展模式  
   从“单届世界杯”拓展到长期执教或多赛事模式。

## 12. 默认决策汇总

为避免后续实现反复讨论，本文档锁定以下默认决策：

- 题材为 `2026 世界杯`
- 玩家扮演 `单支国家队主教练`
- 单局目标为 `带队冲击世界杯冠军`
- 单场交互为 `关键事件回合制`
- 首版事件类别固定为 `临时战术安排 / 特殊换人事件 / 球员需求 / 裁判争议`
- 球队和球员数据口径为 `官方最终名单`
- 数据默认存于 `浏览器内嵌 SQLite`
- 赛果由 `基础实力分 + 对抗修正分` 计算
- 各模块必须解耦，后续升级不依赖重写 UI 或数据层
