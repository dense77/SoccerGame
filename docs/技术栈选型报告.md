# 技术栈选型报告 — SoccerGame

## 项目背景

足球文字游戏，纯前端交互（按钮驱动），前期使用前端小型数据库，后续可能接入外部 API。无需后端服务，静态部署即可运行。

---

## 方案对比

### 方案一：纯 HTML + CSS + Vanilla JS + localStorage

| 维度 | 评价 |
|------|------|
| 上手难度 | 极低，零依赖，一个 HTML 文件就能跑 |
| 开发体验 | 状态管理靠手动 DOM 操作，代码量上去后会非常混乱 |
| 数据存储 | localStorage 5-10MB 上限，只能存字符串，无法做结构化查询 |
| 扩展性 | 差。新功能容易陷入回调地狱，难以维护 |
| 适用场景 | 原型验证、极简小游戏 |

### 方案二：React + Vite + Dexie.js（IndexedDB 封装库）

| 维度 | 评价 |
|------|------|
| 上手难度 | 中等，需要了解组件、hooks、状态管理 |
| 开发体验 | 组件化天然适合游戏 UI（比赛面板、球队管理、事件日志等），状态流向清晰 |
| 数据存储 | Dexie.js 对 IndexedDB 的 Promise 化封装，API 简洁，容量可达数百 MB |
| 扩展性 | 很好。后续加 API 直接用 fetch/axios，加路由用 React Router，无痛升级 |
| 适用场景 | 中大型项目，长期维护，需要灵活扩展 |

### 方案三：Vue 3 + Vite + Pinia + Dexie.js

| 维度 | 评价 |
|------|------|
| 上手难度 | 中等，模板语法对后端开发者更友好 |
| 开发体验 | 响应式系统简洁，单文件组件直观 |
| 数据存储 | 同方案二 |
| 扩展性 | 好。Vue 生态完善，但社区体量略小于 React |
| 适用场景 | 偏好 Vue 的团队，对模板语法更熟悉 |

### 方案四：Svelte + Vite

| 维度 | 评价 |
|------|------|
| 上手难度 | 低，接近原生写法，编译后零运行时 |
| 开发体验 | 代码量少，性能好，但状态管理方案分散 |
| 数据存储 | 需要手动接入 IndexedDB，缺少成熟封装 |
| 扩展性 | 生态较小，遇到问题找解决方案的成本更高 |
| 适用场景 | 追求极致性能、小团队偏好 |

---

## 推荐方案：React + Vite + Dexie.js

### 核心理由

1. **组件化天然匹配游戏结构** — 比赛引擎、球队面板、事件流、按钮交互，每个都是独立可复用组件，状态流向清晰
2. **Dexie.js 是最适合"前端小型数据库"的方案** — 比 localStorage 强大（结构化查询、索引、事务），比 sql.js 轻量（不用加载 3MB WASM），API 简洁直观：
   ```js
   await db.players.add({ name: '梅西', team: 'A队', stats: {...} });
   const allPlayers = await db.players.toArray();
   ```
3. **后续接 API 零成本** — useEffect + fetch 即可，也可后续接入 TanStack Query
4. **全是静态文件，无需后端** — vite build 后可直接部署到 GitHub Pages、Vercel 等静态托管平台
5. **生态最强，踩坑最少** — 动画、音效、国际化、PWA 离线运行等需求，React 社区都有成熟方案

### 不选其他方案的原因

- **纯 JS**：状态复杂度上升后极难维护
- **Vue / Svelte**：本身优秀，但 React 的生态冗余度在这个项目里是优势而非劣势——不需要自己造轮子

---

## 技术栈总览

| 层级 | 技术 | 说明 |
|------|------|------|
| 框架 | React 18+ | UI 组件化 |
| 构建工具 | Vite | 快速热更新，开箱即用 |
| 语言 | TypeScript | 类型安全，减少运行时错误 |
| 数据库 | Dexie.js | 浏览器端 IndexedDB 封装 |
| 样式 | CSS Modules / Tailwind CSS | 根据需要选择 |
| 部署 | GitHub Pages / Vercel | 静态托管，免费 |
