# 世界杯模拟器比赛规则设计

## 1. 文档目标

本文档定义首版比赛规则的输入、计算流程、权重口径和测试基线，用于约束后续 `Match Engine Module` 的实现。

首版原则：

- 结果可解释
- 规则可调参
- 强队有优势但非绝对
- 事件有影响但不直接破坏基本实力逻辑

## 2. 比赛输入

比赛引擎每场至少接收以下输入：

- 我方首发 11 人
- 我方替补名单
- 我方阵型
- 我方战术模板
- 我方球员状态
- 对方首发模板
- 对方阵型
- 对方战术模板
- 对方状态模板
- 当前赛事阶段
- 当前事件池

## 3. 比赛输出

比赛结束后至少输出：

- 最终比分
- 双方基础实力分
- 双方对抗修正分
- 关键事件摘要
- 赛后状态变化
- 用于赛事推进的结果对象

## 4. 比赛计算流程

首版比赛流程固定为：

1. 构建双方赛前上下文
2. 计算基础实力分
3. 计算阵型和战术修正
4. 推进关键事件阶段并写回上下文
5. 计算最终比赛强度差
6. 映射到结果区间和比分
7. 输出赛后快照

## 5. 基础实力分规则

### 5.1 组成项

基础实力分由以下项目构成：

- 首发 11 人综合能力平均值
- 首发位置适配度修正
- 球员体能修正
- 球员情绪修正
- 阵容完整度修正

### 5.2 建议权重

首版建议权重：

- 综合能力：`60%`
- 位置适配：`15%`
- 体能：`10%`
- 情绪：`5%`
- 阵容完整度：`10%`

说明：

- 这是首版默认口径，不代表最终数值平衡版本
- 所有权重应集中定义在规则层，不分散写在 UI 或组件中

### 5.3 阵容完整度规则

阵容完整度用于惩罚明显不合理排阵，例如：

- 核心球员缺阵过多
- 多名球员不在主位置
- 关键位置没有合格人选

首版只做轻量修正，不做复杂协同计算。

## 6. 对抗修正分规则

### 6.1 组成项

对抗修正分由以下项目构成：

- 阵型克制修正
- 战术相性修正
- 事件选择修正
- 临场球员表现修正

### 6.2 阵型克制规则

首版采用标签对抗，不采用空间仿真。

阵型标签示例：

- `wing_play`
- `central_control`
- `high_press`
- `low_block`
- `counter_attack`

处理原则：

- 每套阵型可附带 1 到 3 个标签
- 某些标签对某些标签存在 `-2` 到 `2` 的离散修正
- 阵型克制是修正项，不直接替代基础实力分

### 6.3 战术相性规则

战术模板之间以离散相性表处理：

- 激进进攻 vs 低位防守
- 控球组织 vs 高位逼抢
- 稳守反击 vs 压上进攻

相性结果只提供有限修正，不直接决定比分。

### 6.4 事件修正规则

事件修正由事件系统输出，统一写入比赛上下文。

限制：

- 单个事件不能直接把比赛从极弱势翻成极强势
- 同类事件叠加需要有上限
- 高风险决策可以带来高收益，但也应带来负面波动

## 7. 比分生成原则

### 7.1 结果强度差

比赛引擎在汇总全部修正后得到一个“结果强度差”：

- `result_delta = base_strength_delta + matchup_modifier_delta + event_modifier_delta`

### 7.2 结果区间

首版不追求真实 xG 模拟，采用离散结果区间：

- 明显优势
- 轻微优势
- 均势
- 轻微劣势
- 明显劣势

比分生成原则：

- 均势局面更容易出现平局或小比分
- 明显优势更容易出现 1 球或 2 球胜利
- 极端大比分应低频出现

### 7.3 合理比分池

首版常见比分池建议：

- `0:0`
- `1:0`
- `1:1`
- `2:1`
- `2:0`
- `0:1`
- `1:2`
- `0:2`
- `2:2`
- `3:1`

极端比分如 `4:0`、`0:4` 首版应低频。

## 8. 球员状态规则

首版球员状态至少影响以下内容：

- `available`：无额外修正
- `good_form`：小幅正向修正
- `fatigued`：体能和持续性负向修正
- `suspended`：不可出场
- `injured`：不可出场

情绪类状态可通过独立数值表现，不额外扩展过多枚举。

## 9. 赛后状态变化规则

比赛结束后需要生成：

- 球员体能消耗
- 球员情绪变化
- 停赛状态变化
- 伤病风险结果

生成原则：

- 高强度战术会带来更高体能消耗
- 连续不上场或被拒绝需求的球员可能出现情绪下降
- 激进裁判事件可能放大负面波动

## 10. 参数管理规则

所有比赛参数必须集中管理，不允许散落在页面中。

首版参数至少集中包含：

- 基础实力权重
- 阵型克制表
- 战术相性表
- 事件修正上限
- 比分池和结果区间映射

## 11. 单元测试要求

比赛规则相关单元测试至少覆盖：

- 基础实力分计算
- 阵型克制修正
- 战术相性修正
- 事件修正叠加
- 比分映射规则

建议单元测试用例：

- 用例 1：相同阵容在去掉疲劳和情绪惩罚后，基础实力分上升
- 用例 2：明显不合理排阵会降低阵容完整度分
- 用例 3：存在克制标签时，对抗修正分高于无克制情况
- 用例 4：两个中等强度事件叠加不会超过设定上限
- 用例 5：明显优势局面生成 `0:4` 这类极端比分的概率显著低于常见比分

## 12. 集成测试要求

比赛规则相关集成测试至少覆盖：

- 阵容管理输出能被比赛引擎直接消费
- 事件系统写回的修正能影响最终结果
- 赛后结果对象能被赛事推进模块消费
- 赛后状态变化能被存档模块持久化

建议集成测试用例：

- 用例 1：`Team Management Module -> Match Engine` 联动时，合法阵容可直接进入比赛结算
- 用例 2：`Event System -> Match Engine` 联动时，不同事件选择产生不同结果修正
- 用例 3：`Match Engine -> Tournament Module` 联动时，比赛结果能正确更新积分榜
- 用例 4：`Match Engine -> SaveRepository` 联动时，赛后状态快照可被下一场读取

## 13. 非目标

首版比赛规则当前不实现：

- 逐分钟文本直播
- 真实球员级统计模拟
- 高精度物理或战术空间仿真
- 复杂 AI 教练对策树
